# SPM Bridge: Importing the Existing Skills Ecosystem

How SPM connects to the 200+ skills already published on skills.sh and GitHub.

---

## 1. Landscape

The agent skills ecosystem as of early 2026 has three distribution channels:

**GitHub repos** (primary) — Most skills live as directories inside GitHub repos. Vercel's CLI clones the repo and extracts skill folders. Examples: `vercel-labs/agent-skills` (contains 40+ skills in one repo), `anthropic/skills` (official Anthropic skills), individual author repos.

**skills.sh** — Vercel's leaderboard and directory. Indexes GitHub repos, shows install counts via anonymous telemetry. No registry API — it's a read-only directory that points to GitHub. 200+ skills indexed at launch.

**npm packages** — Vercel's CLI supports `skills install npm:<package>`. Less common but some organizations publish skills as npm packages for private registry support.

All three channels use the same format: a directory with a `SKILL.md` file, optional scripts, and optional metadata. There's no manifest.json, no versioning, no signatures, no security scanning beyond Snyk's submission-time check.

---

## 2. What SPM Adds

| Feature      | skills.sh / GitHub      | SPM                                        |
| ------------ | ----------------------- | ------------------------------------------ |
| Discovery    | Leaderboard only        | Full-text search, categories, filters      |
| Versioning   | None (latest commit)    | Semver, lock files, version pinning        |
| Security     | Snyk scan at index time | 3-layer pipeline (pattern + ML + Lakera)   |
| Trust        | None                    | 4-tier trust system with Sigstore signing  |
| Metadata     | SKILL.md only           | manifest.json + SKILL.md + structured tags |
| Offline      | Requires GitHub access  | .skl packages cached locally               |
| Dependencies | None                    | Skill-to-skill deps (planned)              |
| Reviews      | None                    | User ratings and reviews                   |
| Analytics    | Install count only      | Downloads, weekly trends, agent breakdown  |

The bridge goal: **make it trivial to bring existing skills into SPM** without requiring authors to do anything. Skills that are already good should appear in the SPM registry with minimal friction.

---

## 3. Bridge Strategy

Three modes, from easiest to most complete:

### 3a. Proxy Install (Day 1)

SPM can install directly from GitHub repos, just like the Vercel CLI. No migration needed.

```bash
# Install a skill from a GitHub repo (not in SPM registry)
spm install github:vercel-labs/agent-skills/web-design-guidelines

# Shorthand for well-known repos
spm install github:anthropic/skills/pdf
```

**How it works:**

1. Clone/fetch the repo (sparse checkout for the specific skill directory)
2. Run SPM's 3-layer security scan on the SKILL.md and scripts
3. Install to `~/.spm/skills/<name>/<pseudo-version>/`
4. Link to agents via normal SPM flow
5. No manifest.json generated — treated as `registered` trust, unsigned

**What this gives you:** Immediate access to the entire existing ecosystem without waiting for authors to publish to SPM. Skills pass through SPM's security scanner, which they wouldn't get from raw `npx skills add`.

**What it doesn't give you:** Versioning (always latest commit), no reviews, no categories, no trending data, can't be found via `spm search`.

### 3b. Auto-Import (Week 1 post-launch)

A batch tool that reads skills.sh's public data and creates SPM registry entries for the top skills, credited to a bridge account.

```bash
# One-time bulk import tool (run by SPM admin)
spm-bridge import-from-skillssh --top 100 --dry-run
spm-bridge import-from-skillssh --top 100 --execute
```

**How it works:**

1. **Crawl** — Fetch the skills.sh leaderboard. For each skill, clone the repo and extract the skill directory.

2. **Transform** — Generate a `manifest.json` from the SKILL.md content:

```
GitHub skill directory:
├── SKILL.md
├── scripts/
│   └── analyze.py
└── examples/
    └── demo.csv

    ↓  spm-bridge transform

SPM package:
├── manifest.json      ← generated
├── SKILL.md           ← copied
├── scripts/
│   └── analyze.py     ← copied
└── examples/
    └── demo.csv       ← copied
```

The `manifest.json` is generated by:

- **name**: Extracted from SKILL.md frontmatter `name:` field, or inferred from directory name
- **version**: `0.0.0-bridge.1` (bridge pseudo-version, indicates not author-published)
- **description**: First paragraph of SKILL.md, or frontmatter `description:`
- **category**: LLM-classified from SKILL.md content (same hybrid system as normal publish)
- **tags**: Extracted from SKILL.md keywords, repo topics, skills.sh metadata
- **author**: `bridge/<github-owner>` (e.g., `bridge/vercel-labs`) — clearly not the real author
- **license**: From repo LICENSE file, default `MIT`
- **platforms**: `["all"]`
- **repository**: Original GitHub URL

3. **Scan** — Run all three security scan layers. Block/flag as normal.

4. **Publish** — Publish to SPM registry under the `bridge` namespace. These show up in search with a `🔗 Bridged` badge instead of a trust tier.

5. **Track** — Store mapping: `skill name → github repo + path + commit SHA`. Used for sync.

**Bridged skill display:**

```
📦 web-design-guidelines  0.0.0-bridge.1
🔗 Bridged from vercel-labs/agent-skills
   Originally by: vercel-labs (GitHub)
   ⚠ This skill was auto-imported. The original author can
     claim it by running: spm claim web-design-guidelines
```

### 3c. Author Claim (Ongoing)

When an original author wants to take ownership of their bridged skill:

```bash
spm claim web-design-guidelines
```

**Flow:**

1. Author runs `spm login` (GitHub OAuth)
2. SPM verifies their GitHub identity matches the original repo owner
3. Ownership transfers from `bridge/<owner>` to the real author
4. Author can now publish real versions (1.0.0+), add manifest.json, sign with Sigstore
5. Bridge pseudo-version (`0.0.0-bridge.x`) is deprecated automatically
6. Existing installs get an update notification

This is the smoothest path: the skill already exists in SPM, has downloads and reviews. The author just steps in and takes over.

## 5. Manifest Generation

The bridge generates `manifest.json` from existing SKILL.md content. Here's the mapping:

### From SKILL.md Frontmatter

Many skills use YAML frontmatter in their SKILL.md:

```markdown
---
name: web-design-guidelines
description: Review UI code for compliance with web interface best practices.
---
```

This maps directly:

| SKILL.md source               | manifest.json field      |
| ----------------------------- | ------------------------ |
| `name:` frontmatter           | `name`                   |
| `description:` frontmatter    | `description`            |
| First `#` heading             | `name` (fallback)        |
| First paragraph after heading | `description` (fallback) |

### From Repository Metadata

| Source                          | manifest.json field     |
| ------------------------------- | ----------------------- |
| Repo LICENSE file               | `license`               |
| Repo `package.json` (if exists) | `license`, `repository` |
| GitHub repo topics              | `tags`                  |
| GitHub repo URL                 | `repository`            |
| LLM classification of SKILL.md  | `category`              |
| Directory file listing          | `files`                 |

### Generated Fields

| Field         | Value                                                                                                                                                            |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `version`     | `"0.0.0-bridge.1"`                                                                                                                                               |
| `spm_version` | `">=0.1.0"`                                                                                                                                                      |
| `platforms`   | `["all"]`                                                                                                                                                        |
| `bridge`      | `{ "source": "github", "repo": "vercel-labs/agent-skills", "path": "skills/web-design-guidelines", "commit": "abc1234", "imported_at": "2026-03-01T00:00:00Z" }` |

### Example Generated Manifest

```json
{
  "name": "web-design-guidelines",
  "version": "0.0.0-bridge.1",
  "description": "Review UI code for compliance with web interface best practices. Audits your code for 100+ rules covering accessibility, performance, and UX.",
  "category": "frontend",
  "tags": ["react", "nextjs", "accessibility", "performance", "ux"],
  "author": "bridge/vercel-labs",
  "license": "MIT",
  "platforms": ["all"],
  "repository": "https://github.com/vercel-labs/agent-skills",
  "spm_version": ">=0.1.0",
  "bridge": {
    "source": "github",
    "repo": "vercel-labs/agent-skills",
    "path": "skills/web-design-guidelines",
    "commit": "abc1234def5678",
    "imported_at": "2026-03-01T00:00:00Z"
  }
}
```

## 6. Sync Strategy

Bridged skills need to stay in sync with their GitHub source.

**Daily cron job:**

1. For each bridged skill, fetch the latest commit on the tracked branch
2. If commit SHA changed:
   - Re-clone the skill directory
   - Re-run security scan
   - Bump bridge version: `0.0.0-bridge.1` → `0.0.0-bridge.2`
   - Publish new version to SPM registry
3. If repo is deleted or skill directory removed:
   - Mark bridged skill as deprecated
   - Show "Source repo no longer available" message

**Sync output (admin dashboard):**

```
Bridge sync — 2026-03-01
  ✓ 87 skills checked
  ↑ 4 skills updated (new commits)
  ⚠ 1 skill flagged on re-scan (web-scraper — ML confidence 0.68)
  ✗ 1 skill source gone (old-tool — repo deleted)
```

## 7. Name Collision Handling

What happens when a bridged skill name conflicts with a natively published skill?

**Rules:**

1. Native SPM skills always win. If someone publishes `web-design-guidelines` natively before the bridge imports it, the bridge skips it.
2. If a bridged skill exists and someone publishes the same name natively, the bridge version is deprecated in favor of the native one.
3. The `spm claim` flow handles the common case where the original author wants to take over their bridged skill.
4. Multi-skill repos (like `vercel-labs/agent-skills` with 40+ skills) get individual names, not a monorepo package. Each skill directory becomes its own SPM package.

**Name transformation for multi-skill repos:**

```
vercel-labs/agent-skills
├── skills/
│   ├── web-design-guidelines/     → spm: web-design-guidelines
│   ├── vercel-react-best-practices/ → spm: vercel-react-best-practices
│   ├── react-native-skills/       → spm: react-native-skills
│   └── deploy-to-vercel/          → spm: deploy-to-vercel
```

If the directory name conflicts with an existing SPM package, prefix with the repo owner: `vercel-web-design-guidelines`.

## 8. Display in SPM

Bridged skills look slightly different in search results and detail pages.

**Search result:**

```
  web-design-guidelines  0.0.0-bridge.3    🔗 Bridged
  Review UI code for compliance with web interface best practices
  from: vercel-labs/agent-skills · frontend · ⬇ 340 · ★ —
```

**Detail page differences:**

- "🔗 Bridged from GitHub" banner at top instead of trust badge
- "Claim this skill" CTA for the original author
- Version history shows bridge sync dates, not author publishes
- No Sigstore signature (unsigned)
- Security scan results still shown (SPM scans all bridged skills)
- Link to original GitHub repo prominently displayed

**Install prompt:**

```bash
spm install web-design-guidelines

  ⚠ web-design-guidelines is a bridged skill (not natively published).
    Source: github.com/vercel-labs/agent-skills
    Trust: unverified (no signature, no author claim)

  Install anyway? (Y/n) _
```

Once author claims and publishes 1.0.0+, the bridged warning disappears and it looks like any other SPM skill.

## 9. Reverse Bridge: SPM → Vercel CLI

SPM skills should also be installable via Vercel's `skills` CLI for users who prefer that workflow.

**Option A: GitHub mirror** — SPM maintains a GitHub mirror repo where each published skill gets a directory. The Vercel CLI can install from it:

```bash
npx skills add spm-registry/skills/data-viz
```

**Option B: npm publish** — SPM auto-publishes each skill as an npm package under a scope:

```bash
npx skills install npm:@spm/data-viz
```

**Option C: Vercel CLI plugin** (if Vercel exposes a plugin API) — A Vercel skills plugin that resolves `spm:<name>` to the SPM registry:

```bash
npx skills add spm:data-viz
```

Option B is most realistic at launch since the Vercel CLI already supports npm sources. SPM would auto-publish a minimal npm package for each skill containing just the SKILL.md and scripts directory.

## 10. Database Schema Additions

```sql
-- Bridge tracking table
CREATE TABLE bridge_skills (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  skill_id      UUID NOT NULL REFERENCES skills(id),
  source        TEXT NOT NULL,               -- 'github', 'npm', 'skillssh'
  source_repo   TEXT NOT NULL,               -- 'vercel-labs/agent-skills'
  source_path   TEXT,                        -- 'skills/web-design-guidelines'
  source_commit TEXT,                        -- latest synced commit SHA
  source_branch TEXT DEFAULT 'main',
  last_synced   TIMESTAMPTZ NOT NULL DEFAULT now(),
  sync_status   TEXT DEFAULT 'active',       -- 'active', 'stale', 'gone'
  claimed       BOOLEAN NOT NULL DEFAULT false,
  claimed_by    UUID REFERENCES users(id),
  claimed_at    TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_bridge_source ON bridge_skills(source_repo);
CREATE INDEX idx_bridge_skill ON bridge_skills(skill_id);
```

The `bridge` field in manifest.json is also stored in the `versions.manifest` JSONB column, so it's queryable without joining.

| -------------- | --------------------- | ------------------------------------------------------- |
| Proxy install  | Launch (Week 12)      | `spm install github:...` works, scans on the fly        |
| Bridge tool    | Week 13 (post-launch) | `spm-bridge import` bulk imports top 100 from skills.sh |
| Claim flow     | Week 14               | `spm claim` lets authors take ownership                 |

| Reverse bridge | Week 16+              | Auto-publish to npm for Vercel CLI compatibility        |

This gets the full existing ecosystem into SPM within 2 weeks of launch, with a clear path for authors to take ownership and upgrade to full SPM features (versioning, signing, reviews).

## 12. Risk Mitigation

| Risk                                           | Mitigation                                                                 |
| ---------------------------------------------- | -------------------------------------------------------------------------- |
| Author objects to bridge import                | Immediate takedown on request via `spm bridge remove <name>`               |
| Bridge floods registry with low-quality skills | Only import top 100 by install count; manual review of borderline skills   |
| Name squatting via bridge                      | Bridge names are reservation-locked — only original author can claim       |
| skills.sh changes structure                    | Bridge tool is scraping-resilient; falls back to direct GitHub clone       |
| Vercel adds their own registry API             | SPM can integrate as an upstream source, not compete directly              |
| License issues                                 | Bridge preserves original license; adds `bridge` metadata for transparency |
